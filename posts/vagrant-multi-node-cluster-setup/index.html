Multi-Node Cluster Setup using Vagrant

Posted Jan 25, 2017 2017-01-25T08:00:00+03:00 by S. Aya
Updated May 2, 2020 2020-05-02T17:19:33+03:00

Introduction

For most small scale applications a single-node cluster should suffice. But, for relatively large scale, distributed applications a multi-node cluster is more suitable. To achieve this, one option is to build real physical servers (on-premise) for cluster environment. The clustered servers (nodes) are connected by physical cables and by software. Creating multiple nodes, wiring, networking and complex software installations take a lot of time. Another option is using the cloud; a revolutionary movement founded on the idea of cheap, disposable computing resources. Let’s think about Hadoop ecosystem: Hadoop pseudo distributed mode or local model in single node is an option to get started. However, as mentioned in the beginning, what if we want to test our hadoop/spark program, hive or pig scripts in the real hadoop distributed mode? What if we don’t want to rent out cloud services which we cannot afford with incurring cost. We have a quick solution called Vagrant, which is a tool for building complete development environments. Vagrant takes the idea of the cloud, and makes this idea available on anyone’s desktop. With just two commands and zero configuration, Vagrant automatically builds and configures a fully featured virtual machine for any purpose.

Test system: MacBoook Pro (Late 2013) 2.3 GHz Intel Core i7, 16 GB RAM, 512 GB SSD.

  • macOS Sierra (Version 10.12.3)
  • VirtualBox (Version 5.0.32 r112930)
  • Vagrant 1.9.1

Walk-through of an example Vagrantfile

Using the following Vagrantfile, simply type vagrant up in the command line and then you’ll have a running cluster (linux) with two VMs with the specified properties (CPU, RAM, IP Address, etc.). Then, vagrant ssh into one of the nodes and test the connection with ping. When you’re done, type vagrant destroy to shutdown VMs.

What just happened?

Following Vagrantfile simply creates a virtual machine (server, node), “cldemo-template” is the label of the node “demo_config” is the variable in the second line and in the third line we see that variable (that machine) is configured with the box named “bento/centos-6.7”.

Vagrant.configure("2") do |config|
    config.vm.define "cldemo-template" do |demo_config|
        demo_config.vm.box = "bento/centos-6.7"
    end
end

The repository is a folder sitting on the host machine, and is used for downloading binaries from host into vm and in the vm install the software, downloading the zip, tar files etc.

varDomain = "demo-c.internal"

There are many images hosted in hashicorp repository. Here, we download and then use the bento/centos-6.7 image.

The following section allows all nodes to see each other, and talk to each other with the vagrant plugin host manager.

config.hostmanager.enabled = true
config.hostmanager.manage_host = true
config.hostmanager.manage_guest = true
config.hostmanager.ignore_private_ip = false
config.hostmanager.include_offline = true

To install a software, we use the following external script. The shell provisioner specifies the path to a shell script on the host machine. Vagrant will then upload this script into the guest and execute it.

1

demo_config.vm.provision :shell, :path => "java/provision_for_java.sh"

The following line is very important. To install a software for a specific server(node):

if demo_node[:host] == "cldemo-template"
    demo_config.vm.provision :shell, :path => "java/provision_for_maven"
end

What else?

Different operating systems, vagrant or virtualbox versions might cause problems.
